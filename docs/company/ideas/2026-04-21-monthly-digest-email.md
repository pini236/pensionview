## The idea

Send a monthly "your pension changed this month" email to every user when a new report is processed, so the product creates its own re-engagement trigger instead of relying on users to remember to open it.

## Why it matters

The Gmail pipeline already watches for new PDFs and processes them automatically. The product knows the moment new data lands. But then — nothing. No notification reaches the user. They have to happen to open the app to see "your savings went up ₪4,200 this month" or "your employer missed a deposit."

Israeli pension reports arrive monthly. That's the natural cadence. If we send one email per processed report and that email contains a concrete number ("your total savings: ₪847,300 — up ₪4,200 from last month"), we're creating a monthly re-engagement loop that maps exactly to the real-world event that drives value.

Targeted metrics:
- **Monthly active users** (currently unmeasured — the key retention metric for a monthly-cadence product).
- **Deposit alert click-through** — if there's an alert in the digest, click-to-dashboard should be high because the cost is real and visible.

Without this, retention depends entirely on intrinsic motivation. That's a bet we shouldn't make.

## How it might work

- Trigger: when a report transitions to `status = 'done'` in the processing queue, enqueue a digest email for that `profile_id`.
- Content (data already in Supabase):
  - Total savings vs. prior report (absolute + %).
  - Any deposit alerts (severity, fund name, month missing).
  - Fee savings potential if non-zero.
  - One sentence from `report_insights.summary_text` (already generated by the AI pipeline).
  - CTA: "See the full breakdown" → `/dashboard`.
- Send via Resend (or whichever transactional provider is already in the stack — check with engineering-lead before assuming). Subject line in Hebrew if profile locale is `he`.
- Respect: if the user has no `google_access_token` (they upload manually), fire the email on `status = 'done'` the same way.
- One email per report. No separate digest cadence to manage.

## What we don't know

- Whether a transactional email provider is already set up (no evidence in the codebase — this may add infra cost; needs engineering-lead sign-off).
- Opt-out mechanics: Israeli CPA law requires unsubscribe. Trust-compliance-lead must review before shipping.
- Whether users want email from us at all — there's no consent signal in the current schema. We may need to add an `email_digest_opt_in` column and default it to true for new signups (with disclosure at signup).
- Whether the Gmail-connected flow creates an implicit expectation of email (it shouldn't — Gmail OAuth is for reading PDFs, not for sending to the user's address).
- Deliverability: sending from a pensionview.* domain to Israeli Gmail users requires SPF/DKIM/DMARC setup.
