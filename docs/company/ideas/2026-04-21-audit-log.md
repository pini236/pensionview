## The idea

Introduce a tamper-evident audit log that records every access and mutation of pension PII so we can answer "who saw what, and when" in the event of a complaint, incident, or regulator inquiry.

## Why it matters

PensionView holds pension balances, national IDs (encrypted at rest), date of birth, insurance coverages, and Google OAuth tokens. Every interaction with this data currently leaves no durable record outside of ephemeral Vercel log lines.

**What we cannot answer today:**
- "Did anyone access my pension data between date X and date Y?" (required by חוק הגנת הפרטיות s.17 right of inspection)
- "Was national_id ever decrypted and for what purpose?" (the decrypt step in `lib/pipeline/decrypt/route.ts:51` unwraps the AES ciphertext to use as a PDF password — no record of this event exists beyond a console.log in the pipeline step)
- "Which AI advisor queries included my full financial snapshot?" (every call to `app/api/advisor/chat/route.ts:29-59` loads savings, insurance, and member profiles into an Anthropic API request — no event is recorded)
- "Who modified a household member's national_id?" (PATCH `/api/members/[id]` at `app/api/members/[id]/route.ts:193` updates PII with no event log)

This is not just a regulatory concern. It is our primary incident-response tool. If a member reports that their data "seems wrong" or alleges a leak, we have no forensic trail. We are flying blind.

The Privacy Protection Regulations (Data Security), 5777-2017, Regulation 8 requires that an information system maintain access logs for data classified as "sensitive" — pension data is explicitly in the highest sensitivity tier under Israeli law.

OWASP A09:2021 (Security Logging and Monitoring Failures) identifies this exact gap as a top-10 web application security risk.

## How it might work

- New Supabase table `audit_events`:
  ```
  id              uuid primary key default gen_random_uuid()
  event_type      text not null   -- 'pii_read' | 'pii_write' | 'ai_query' | 'pipeline_decrypt' | 'export' | 'deletion'
  actor_email     text            -- authenticated user's email; null for pipeline/service-role events
  target_type     text            -- 'profile' | 'report' | 'savings_product' | 'insurance_product'
  target_id       uuid
  household_id    uuid
  metadata        jsonb           -- e.g. { fields_changed: ['national_id'], pipeline_step: 'decrypt' }
  created_at      timestamptz not null default now()
  ```
- RLS on `audit_events`: INSERT only for service_role; SELECT for authenticated users scoped to their own `household_id` (so members can inspect their own access log via the export endpoint).
- Instrument the four highest-priority paths first (ordered by sensitivity):
  1. `lib/pipeline/decrypt/route.ts` — log `pipeline_decrypt` event with `reportId`, no raw national_id in metadata
  2. `app/api/advisor/chat/route.ts` — log `ai_query` event with `household_id`, query length (not content), member count included in context
  3. `app/api/members/[id]/route.ts` PATCH — log `pii_write` with `fields_changed` array
  4. `app/api/account/export` (future) — log `export` event
- `logEvent()` in `lib/observability.ts` already emits structured JSON to Vercel logs — audit events are different: they go to the DB so they survive log rotation and are queryable per-member.
- Retention: 2 years (matches Israeli data security regulation guidance; delete older rows via a monthly cron).

## What we don't know

- Write volume: if the AI advisor sees heavy use, `ai_query` events could be high-frequency. Consider sampling (log 1 in N if volume exceeds threshold) vs. logging all and using a partitioned table.
- Whether to surface the audit log to members in the UI (a "who accessed your data" view) — valuable trust signal, but scope creep for v1. Export endpoint is the minimum viable disclosure.
- Service-role pipeline events have no `actor_email` — we need a `pipeline_run_id` or similar to correlate events across a single pipeline execution.
